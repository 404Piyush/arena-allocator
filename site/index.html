A bump arena memory allocator in C11.

O(1) arena_alloc. O(1) arena_reset. 16-byte aligned. Single mmap. ~14× faster than malloc for small allocations. 143 assertions, all passing.

Why a bump arena?

O(1) every op

No per-allocation bookkeeping. arena_alloc is ~4 instructions: align-up, bounds check, bump pointer, return.

No fragmentation

Used memory is always contiguous. There's no free list to coalesce, no first-fit heuristics, no size class buckets.

The phase pattern

Reset at the end of each request. Perfect for parsers, compilers, request handlers, frame allocators.

API at a glance

The full surface area is six functions:

arena *arena_create (size_t capacity);
void  *arena_alloc   (arena *a, size_t size);
void   arena_reset   (arena *a);
void   arena_destroy (arena *a);
size_t arena_in_use  (const arena *a);
size_t arena_high_wat(const arena *a);

The phase pattern

arena *a = arena_create(64 * 1024);
for (;;) {                           /* per-request loop */
    arena_reset(a);                  /* one pointer write   */
    request_t *r = arena_alloc(a, sizeof *r);
    r->method = arena_alloc(a, strlen(line) + 1);
    r->body   = arena_alloc(a, body_len);
    handle(r);
}
arena_destroy(a);

The cost of memory management per request collapses from N frees to one reset.

When to use a bump arena

Use it for any workload with a clear phase: build a tree of allocations, use it, throw it all away, repeat. Compilers, parsers, request handlers, frame allocators in a renderer, simulation loops — all fit the pattern.

Do not use it when individual allocations have lifetimes that don't share a common destruction point. That's what malloc is for.
